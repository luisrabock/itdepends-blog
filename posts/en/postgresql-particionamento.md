Every application has that table. The one that grew alongside the business, received column after column over the years, became the reference for half the data model, and one day started making everything slow.

I worked on a system that had exactly that table. And the process of solving this with PostgreSQL partitioning was one of the most impactful database tasks I've ever done.

---

## the scenario

The table was central to the operation. I'll call it `device_records` — records continuously generated by devices from multiple companies. Over time, it accumulated **billions of rows**, dozens of columns, and was related to practically every relevant table in the system.

A simplified view of what it stored:

```sql
CREATE TABLE device_records (
    id            BIGSERIAL PRIMARY KEY,
    company_id    BIGINT NOT NULL,
    device_id     BIGINT NOT NULL,
    record_type   VARCHAR(50) NOT NULL,
    status        VARCHAR(30) NOT NULL,
    payload       JSONB,
    processed_at  TIMESTAMP,
    created_at    TIMESTAMP NOT NULL DEFAULT NOW(),
    -- ... ~30 more operational data columns
    FOREIGN KEY (company_id) REFERENCES companies(id),
    FOREIGN KEY (device_id)  REFERENCES devices(id)
);
```

Every application process went through it: reports, async processing jobs, real-time dashboards, partner integrations, cleanup routines. Practically every query had a `WHERE company_id = ?` — it was a multi-tenant system after all.

---

## the symptoms

Degradation was gradual, as it always is. There's no single day when everything stops working. There's a day when the team realizes that what was fast is now slow, and what was slow now hangs.

The first signs:

- Reports that took 3 seconds started taking 30
- Nightly jobs that finished in 2 hours stopped finishing before the next cycle
- Dashboard queries started competing for locks with processing jobs
- `autovacuum` could barely keep up with the write volume

Monitoring showed queries that previously cost a few milliseconds now running with completely different execution plans. PostgreSQL had abandoned the indexes and was doing **sequential scans** on the entire table.

---

## the diagnosis

The first step was understanding what was happening in the execution plans. With `EXPLAIN (ANALYZE, BUFFERS)`:

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, device_id, status, created_at
FROM device_records
WHERE company_id = 42
  AND status = 'PENDING'
  AND created_at >= NOW() - INTERVAL '7 days'
ORDER BY created_at DESC
LIMIT 100;
```

The result looked something like this:

```
Limit  (cost=0.00..98432.11 rows=100 width=48)
       (actual time=45231.442..45231.501 rows=100 loops=1)
  ->  Seq Scan on device_records
        (cost=0.00..2847392.00 rows=289321 width=48)
        (actual time=0.021..45218.332 rows=100 loops=1)
        Filter: (company_id = 42 AND status = 'PENDING' AND ...)
        Rows Removed by Filter: 2847291342
  Buffers: shared hit=12483 read=1847291
Planning Time: 2.3 ms
Execution Time: 45231.8 ms
```

45 seconds. Removing **2.8 billion** rows through the filter. The composite index on `(company_id, status, created_at)` existed — but with a table this large and this write volume, PostgreSQL had decided the cost of using it was higher than doing the full scan.

The problem wasn't missing indexes. It was the absolute size of the table.

---

## why partitioning

Partitioning solves exactly this problem: instead of one table with 3 billion rows, you have N smaller tables (the partitions), and PostgreSQL knows which one each record belongs to.

When a query has `WHERE company_id = 42`, the database executes only against that company's partition — **completely ignoring every other partition**. This mechanism is called **partition pruning**.

Choosing the partition key is the most critical decision. In our case, it was obvious: `company_id`. For two reasons:

1. **Every application query had `WHERE company_id = ?`** — the filter was already there, partitioning would just make it far more efficient
2. **Each company's data is isolated** — no process needed to cross data between companies in the same query

---

## implementing the partitioning

PostgreSQL supports three types of partitioning:

- `RANGE` — by value ranges (ideal for dates)
- `LIST` — by specific values (ideal for discrete IDs like `company_id`)
- `HASH` — uniform distribution by hash (when there's no clear business key)

For `company_id`, we used `LIST`.

**1. Creating the partitioned table:**

```sql
CREATE TABLE device_records_partitioned (
    id            BIGSERIAL,
    company_id    BIGINT NOT NULL,
    device_id     BIGINT NOT NULL,
    record_type   VARCHAR(50) NOT NULL,
    status        VARCHAR(30) NOT NULL,
    payload       JSONB,
    processed_at  TIMESTAMP,
    created_at    TIMESTAMP NOT NULL DEFAULT NOW()
    -- remaining columns
) PARTITION BY LIST (company_id);
```

**2. Creating partitions per company:**

```sql
CREATE TABLE device_records_company_1
    PARTITION OF device_records_partitioned
    FOR VALUES IN (1);

CREATE TABLE device_records_company_2
    PARTITION OF device_records_partitioned
    FOR VALUES IN (2);

CREATE TABLE device_records_company_42
    PARTITION OF device_records_partitioned
    FOR VALUES IN (42);

-- default partition for new companies or unmapped values
CREATE TABLE device_records_default
    PARTITION OF device_records_partitioned
    DEFAULT;
```

**3. Creating indexes on partitions:**

In PostgreSQL, you can create indexes on the parent table and they propagate automatically to all partitions:

```sql
CREATE INDEX idx_dr_company_status_created
    ON device_records_partitioned (company_id, status, created_at DESC);

CREATE INDEX idx_dr_device_created
    ON device_records_partitioned (device_id, created_at DESC);
```

---

## migrating the data

Migrating billions of rows without downtime requires care. The strategy we followed:

**1. Batch migration with the application running:**

```sql
-- Migrate in batches of 100k, prioritizing recent data
INSERT INTO device_records_partitioned
SELECT * FROM device_records
WHERE id BETWEEN :start AND :end
  AND NOT EXISTS (
    SELECT 1 FROM device_records_partitioned p WHERE p.id = device_records.id
  );
```

We ran this process off-peak hours, in batches, over several days.

**2. Final cutover with minimal window:**

Once the historical data was migrated, we did a quick lock on the original table to migrate only the delta generated during the process:

```sql
BEGIN;
LOCK TABLE device_records IN EXCLUSIVE MODE;

-- migrate the final delta
INSERT INTO device_records_partitioned
SELECT * FROM device_records dr
WHERE NOT EXISTS (
    SELECT 1 FROM device_records_partitioned p WHERE p.id = dr.id
);

-- atomic rename
ALTER TABLE device_records RENAME TO device_records_old;
ALTER TABLE device_records_partitioned RENAME TO device_records;

COMMIT;
```

The lock lasted seconds — just the time to migrate the final delta.

## the results

The same query that took 45 seconds:

```
Limit  (cost=0.00..142.31 rows=100 width=48)
       (actual time=1.823..1.891 rows=100 loops=1)
  ->  Index Scan using idx_dr_company_status_created
        on device_records_company_42
        (cost=0.56..142.31 rows=100 width=48)
        (actual time=1.821..1.883 rows=100 loops=1)
        Index Cond: (company_id = 42 AND status = 'PENDING' AND ...)
  Buffers: shared hit=8 read=3
Planning Time: 1.1 ms
Execution Time: 1.9 ms
```

**From 45 seconds to under 2 milliseconds.**

PostgreSQL now goes directly to the `device_records_company_42` partition, which held a fraction of the original data. The index became efficient because the partition was manageable.

The cascading effects were immediate:

- Nightly jobs returned to finishing within their cycles
- Reports returned to their original response times
- `autovacuum` started working properly — smaller partitions are processed faster
- Concurrency locks reduced dramatically, as different companies' operations stopped competing for the same physical space
- Smaller indexes per partition fit better in memory (`shared_buffers`), increasing cache hit rate

## additional gains from partitioning

**Per-company maintenance**

Need to archive or delete all data for a company? Instead of a `DELETE` that scans billions of rows:

```sql
-- Instant partition drop — no I/O cost
DROP TABLE device_records_company_99;

-- Or detach without deleting (for archiving)
ALTER TABLE device_records
    DETACH PARTITION device_records_company_99;
```

**Combined partitioning**

For companies with very high volume, you can sub-partition by date within the company partition:

```sql
CREATE TABLE device_records_company_1
    PARTITION OF device_records_partitioned
    FOR VALUES IN (1)
    PARTITION BY RANGE (created_at);

CREATE TABLE device_records_company_1_2024
    PARTITION OF device_records_company_1
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
```

**Lifecycle policy enforcement**

Remember the post about lifecycle policies? With date-based partitioning, deleting old records becomes a `DROP TABLE` — instant, lock-free, no subsequent vacuum cost.

---

## what to check before going to partitioning

Partitioning isn't the first answer to any performance problem. Before reaching for it, verify:

- Are the correct indexes created and being used?
- Is the query well-written? (no functions on indexed columns, no unnecessary `SELECT *`)
- Is `autovacuum` healthy? Tables with too much dead tuple degrade the planner
- Are PostgreSQL settings (`work_mem`, `shared_buffers`, `effective_cache_size`) adequate for the workload?

Partitioning is for when the table is simply too large for indexes and vacuum to work well on their own — which typically starts happening in the hundreds of millions to billions of records range, depending on hardware.

---

## summary

PostgreSQL partitioning isn't an obscure or niche feature. It's a maturity tool for tables that have grown beyond what indexes alone can handle.

The key to success here was that the partition key (`company_id`) was already present in every application query. The database just needed to know it could use it to eliminate 99% of the data before even starting the search.

When the table is the bottleneck, tuning the query isn't enough. You need to rethink how the data is organized.
