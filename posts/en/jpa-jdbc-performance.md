JPA is one of the most powerful abstractions in the Java ecosystem. It eliminates boilerplate, maps objects to tables declaratively, and lets developers focus on business logic. In growing projects, it significantly speeds up development.

It's also one of the main sources of performance problems in large applications.

These aren't mutually exclusive. Both statements are true at the same time.

---

## what JPA does for you

Before talking about the problems, it's worth acknowledging the value.

Without JPA, every database operation involves manual SQL, mapping `ResultSet` to objects, connection management, and exception handling. JPA abstracts all of that. You define the entity once and the framework handles the rest:

```java
@Entity
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    private Customer customer;

    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL)
    private List<OrderItem> items;

    private BigDecimal total;
    private LocalDateTime createdAt;
}
```

Operations that used to require many lines of code become one:

```java
orderRepository.findById(id);
orderRepository.save(order);
orderRepository.findByCustomerAndCreatedAtAfter(customer, date);
```

In new projects or teams that need delivery speed, this productivity is real and valuable.

## where the problems begin

The problem starts when the developer stops thinking in SQL and starts thinking only in objects.

JPA doesn't eliminate the database — it just hides it. And everything that's hidden still exists and still has a cost. The difference is that now the cost is no longer visible.

### the N+1 problem

The classic one. You fetch a list of orders and iterate over them to access the customers:

```java
List<Order> orders = orderRepository.findAll(); // 1 query
for (Order order : orders) {
    System.out.println(order.getCustomer().getName()); // N queries
}
```

With 500 orders, you just executed 501 queries. With `FetchType.LAZY` (the correct default), each access to an uninitialized relationship triggers a new round trip to the database.

Hibernate even warns about this in logs — but in production, with SQL logging disabled, the problem becomes completely invisible until the application starts hanging.

### unnecessary data loading

Entities mapped with all table columns load everything, always. If you have a `Product` table with 40 columns — including long descriptions, presentation HTML, audit fields, and historical data — every query using that entity loads all 40 columns, even when you only need `id` and `name`.

```java
// You want this:
SELECT id, name FROM product WHERE category_id = ?

// JPA delivers this:
SELECT id, name, description, html_content, audit_user, audit_date,
       historical_price, sku, weight, dimensions, ... FROM product WHERE category_id = ?
```

On tables with `TEXT` or `BLOB` columns, the impact is significant.

### the silent cartesian product

When you map multiple `@OneToMany` and use `JOIN FETCH` to avoid the N+1, Hibernate can generate a cartesian product:

```java
// Seemingly harmless
@Query("SELECT o FROM Order o JOIN FETCH o.items JOIN FETCH o.payments")
List<Order> findAllWithDetails();
```

If an order has 10 items and 3 payments, this query returns 30 rows to represent 1 order. With a thousand orders, the situation scales rapidly. Hibernate deduplicates in memory — but the database already processed and transferred all the rows.

### cascade without criteria

`cascade = CascadeType.ALL` on a relationship looks convenient. In practice, it can mean that saving a parent entity triggers unnecessary updates on dozens of child entities, even when none were modified. In a monolith with many relationships, the SQL trail generated by a single operation can be surprising.

## problems that come from database structure

JPA doesn't fix — and can mask — problems rooted in the database structure.

**Missing indexes**

JPA generates queries with `WHERE`, `JOIN`, and `ORDER BY`. If the involved columns don't have indexes, the database performs a full table scan. On tables with millions of records, the difference between an indexed and non-indexed query can be milliseconds vs minutes.

JPA doesn't create the right indexes automatically. You need to understand the access pattern and create the correct indexes.

```java
// This query runs hundreds of times per minute
orderRepository.findByStatusAndCreatedAtAfter(status, date);

// The database needs:
CREATE INDEX idx_order_status_created ON orders(status, created_at);
```

**Tables without design intent**

A table with 80 columns where half are `nullable`, with generic names like `field1`, `value_aux`, `flag3`, usually indicates that the data model grew without intention. JPA will faithfully map all of it — and the consequences show up in performance, clarity, and maintainability.

**Missing partitioning**

High-volume tables (logs, events, historical records) without partitioning cause queries to degrade over time. JPA doesn't know this and doesn't warn. The query that worked fine with 1 million records starts hanging at 500 million.

## what JDBC still does better

JDBC is often treated as "the old way." It isn't. It's the most direct database access layer in Java, and in specific situations, it's the right answer.

With JDBC (or Spring's `JdbcTemplate`), you write exactly the SQL you need:

```java
// Complex report with aggregations, CTEs, and window functions
String sql = """
    WITH monthly_totals AS (
        SELECT customer_id,
               DATE_TRUNC('month', created_at) AS month,
               SUM(total) AS monthly_total,
               COUNT(*) AS order_count
        FROM orders
        WHERE created_at >= :startDate
        GROUP BY customer_id, DATE_TRUNC('month', created_at)
    )
    SELECT c.name,
           mt.month,
           mt.monthly_total,
           mt.order_count,
           RANK() OVER (PARTITION BY mt.month ORDER BY mt.monthly_total DESC) AS rank
    FROM monthly_totals mt
    JOIN customers c ON c.id = mt.customer_id
    ORDER BY mt.month, rank
    """;

return jdbcTemplate.query(sql, params, rowMapper);
```

JPA can't express this cleanly. You could use `@Query` with a native query, but then you're writing SQL anyway — just with Hibernate's overhead on top.

**Specific projections**

When you need only a few fields, JDBC delivers exactly what you ask for:

```java
// Lightweight fetch for listing
String sql = "SELECT id, name, status, created_at FROM orders WHERE customer_id = ?";
return jdbcTemplate.query(sql, (rs, row) -> new OrderSummary(
    rs.getLong("id"),
    rs.getString("name"),
    rs.getString("status"),
    rs.getTimestamp("created_at").toLocalDateTime()
), customerId);
```

No entity loaded, no unnecessary columns, no relationships resolved.

## the strategy that works in large systems

In large monoliths, the healthiest approach isn't choosing between JPA and JDBC — it's using each where it makes sense.

**Use JPA for:**
- Write operations (insert, update, delete) where entity mapping has value
- Simple queries by ID or direct fields
- Where the domain model is stable and well-defined

**Use JDBC / native queries for:**
- Reports and analytical queries
- Listings with specific projections
- Any query involving aggregations, CTEs, window functions
- Situations where N+1 is unavoidable with JPA
- Batch operations (bulk updates, bulk deletes)

```java
// Repository using both approaches in the same place
@Repository
public class OrderRepository {

    private final JpaRepository<Order, Long> jpaRepo;
    private final JdbcTemplate jdbc;

    // JPA for domain operations
    public Order save(Order order) {
        return jpaRepo.save(order);
    }

    public Optional<Order> findById(Long id) {
        return jpaRepo.findById(id);
    }

    // JDBC for analytical queries
    public List<OrderSummary> findSummariesByCustomer(Long customerId) {
        return jdbc.query(
            "SELECT id, total, status, created_at FROM orders WHERE customer_id = ?",
            (rs, row) -> mapToSummary(rs),
            customerId
        );
    }
}
```

## questions to ask before reaching for JPA

Before throwing a query into Spring Data's `findBy` and moving on, it's worth asking:

1. **What SQL does this generate?** — enable `spring.jpa.show-sql=true` in development and read the queries
2. **How many queries are executed for this use case?** — use an APM or Hibernate's own log
3. **Does the table have the right indexes for this access pattern?** — verify with `EXPLAIN`
4. **Do I need all entity fields?** — if not, use projections or JDBC
5. **Is this data read frequently?** — if so, consider cache before optimizing the query

---

## summary

JPA is a powerful tool that should be used with awareness of what it generates underneath. In small and medium projects, the productivity it offers outweighs the risks. In large systems with high-volume tables and complex queries, ignoring the generated SQL is accumulating performance technical debt.

JDBC isn't primitive. It's direct. And direct, in databases, is often faster, more readable, and more predictable.

The choice doesn't have to be binary. A mature system uses both tools in the right place.
